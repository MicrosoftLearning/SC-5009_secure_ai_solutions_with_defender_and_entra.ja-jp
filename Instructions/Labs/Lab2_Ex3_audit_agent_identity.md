# ラボ 2 - 演習 3 - AI エージェントのアクセス権を確認して監査する

組織の AI エージェント ID は、API アクセス許可と Azure RBAC ロールで構成されています。 エージェントを運用環境に移行する前に、その構成が正しいことを確認し、アクセス権の変更が記録されていることを確認し、資格情報を安全にローテーションできるようにする必要があります。 ID 管理者は、エンタープライズ アプリケーションのアクセス許可を確認し、ロールの割り当てアクティビティを監査し、資格情報のローテーションを実践します。

**タスク**:

1. エージェントの API アクセス許可と同意の状態を確認する
1. エンタープライズ アプリケーション プロパティを確認する
1. ロールの割り当てについて Azure アクティビティ ログを確認する
1. AI エージェントのクライアント シークレットをローテーションする

## タスク 1 – エージェントの API アクセス許可と同意の状態を確認する

アプリケーション ID が API アクセス許可付きで構成されている場合、それらのアクセス許可はエンタープライズ アプリケーションのアクセス許可ブレードに表示されます。 このタスクでは、AI エージェントのアクセス許可が正しく構成されていることを確認し、同意の状態を確認します。

1. **Microsoft Edge** を開き、`https://entra.microsoft.com/` に移動します。

1. ラボ ホスト プロバイダーによって提供されるユーザー名とパスワードを使用してサインインします。

1. 左側のサイド バーの **[Entra ID]** で、**[アプリケーション]** を展開し、**[エンタープライズ アプリケーション]** を選択します。

1. 検索フィールドに「`Contoso-AI-Agent`」と入力し、アプリケーションを選択します。

1. 左側のサイド バーで **[セキュリティ]** を展開し、**[アクセス許可]** を選択します。

1. **[アクセス許可]** ページで、一覧表示されているアクセス許可を確認します。 **[Microsoft Graph]** の下に **User.Read.All**が表示されます。同意の状態は **[未付与]** となっています。

    > **注**: "未付与" 状態は、このアクセス許可が構成されているが、まだアクティブではないことを意味します。 運用環境では、エージェントがこのアクセス許可を使用する前に、全体管理者が管理者の同意を確認して付与します。 この同意ワークフローは重要なセキュリティ チェックポイントであり、アプリケーションが明示的な承認なしにデータにアクセスするのを防ぎます。

1. ここに記載されているアクセス許可と、Azure portal で割り当てられた RBAC ロールの違いに注意してください。 API アクセス許可 (ここに示されている) は、API を通じてデータへのアクセス権を制御します。 (個々の Azure リソースに割り当てられた) RBAC ロールは、Azure インフラストラクチャへのアクセス権を制御します。

AI エージェントの API アクセス許可の構成と同意の状態が正常に確認されました。

## タスク 2 – エンタープライズ アプリケーションのプロパティを確認する

エンタープライズ アプリケーションのプロパティは、テナントでのサービス プリンシパルの動作を制御します。 このタスクでは、エージェントのサインイン動作とアクセス権の制御に影響する主要な設定を確認します。

1. 左側のサイド バーで **[管理]** を展開し、**[プロパティ]** を選択します。

1. **[プロパティ]** ページで、次の設定を確認します。

   - **ユーザーのサインインが有効**: はい
   - **割り当てが必要**: いいえ

    > **注**: **[割り当てが必要]** が **[いいえ]** に設定されている場合、テナント内のすべてのユーザーまたはサービスが、このアプリケーションのトークンを取得できます。 運用環境で、これを **[はい]** に設定すると、トークンの発行は明示的に割り当てられたユーザーまたはグループのみに制限されます。これは、エージェントとして動作する、またはエージェントを介して動作する ID を制限するための重要な制御です。

1. この構成で AI エージェントの無制限のトークン発行が許可されるかどうか、および **[割り当てが必要]** を **[はい]** に変更するかどうかを検討します。

AI エージェントのエンタープライズ アプリケーションのプロパティを正常に確認できました。

## タスク 3 – ロールの割り当てについて Azure アクティビティ ログを確認する

Azure アクティビティ ログは、ロールの割り当てを含む Azure リソースに対する管理操作をキャプチャします。 このタスクでは、AI エージェントのサービス プリンシパルに付与された RBAC ロールの割り当てが記録されたことを確認します。

1. 新しいタブを開き、`https://portal.azure.com` に移動します。

1. Azure portal の上部にある検索バーに「`Activity log`」と入力し、結果から **[アクティビティ ログ]** を選択します。

1. **[アクティビティ ログ]** ページで、**[タイムスパン]** フィルターを **[過去 1 時間]** に設定します。

1. **[操作]** フィルターで、`Create role assignment` を検索して選択し、**[適用]** を選択します。

1. ログ エントリを確認します。 Foundry リソースと Key Vault に対するロールの割り当て操作のエントリが表示されます。

1. **[ロールの割り当ての作成]** エントリのいずれかを選択して、その詳細を表示します。

1. **[要約]** タブを確認し、リソース、操作、状態を確認します。

    > **注**: API アクセス許可を使用して、ID がアクセスできるデータを把握し、アクティビティ ログを使用して実行したアクションを把握します。 これらを組み合わせることで、アプリケーション ID のアクセス権と動作を完全に可視化できます。

AI エージェント ID へのロールの割り当てが Azure アクティビティ ログに記録されていることを正常に確認できました。

## タスク 4 – AI エージェントのクライアント シークレットをローテーションする

クライアント シークレットの有効期限が切れ、侵害される可能性があります。 いずれの場合も、エージェントの操作を中断せずに資格情報をローテーションする必要があります。 このタスクでは、新しいクライアント シークレットを作成し、重複期間中に両方のシークレットがアクティブであることを確認してから、推奨されるローテーション ワークフローに従って古いシークレットを削除します。

1. `https://entra.microsoft.com/` に戻ります。

1. 左側のサイド バーの **[Entra ID]** で **[アプリケーション]** を展開し、**[アプリの登録]** を選択します。

1. **Contoso-AI-Agent** を選択します。

1. 左側のサイド バーで、**[管理]** の **[証明書とシークレット]** を選択します。

1. **[クライアント シークレット]** タブで、既存のシークレット (**AI エージェント認証**) が表示されたことを確認します。

1. **[+ 新しいクライアント シークレット]** を選択します。

1. **[クライアント シークレットの追加]** ポップアップで次のように入力します。

   - **説明**: `Rotated agent credential`
   - **有効期限**: **180 日 (6 か月)**

1. **[追加]** を選択します。

1. 両方のシークレット (元のシークレットと新しく作成されたシークレット) が一覧表示されていることを確認します。

    > **注**: 運用ローテーション中は、この重複期間が重要です。 古い資格情報が削除される前に、新しい資格情報をすべてのコンシューマー (エージェントのホスティング環境、Key Vault、デプロイ パイプライン) に配布する必要があります。 古いシークレットを途中で削除すると、認証が中断されます。

1. 元の **AI エージェント認証**シークレットで、削除アイコンを選択し、削除を確認します。

1. **ローテーションされたエージェント資格情報**シークレットのみが残っていることを確認します。

推奨されるワークフロー (新規作成、重複の確認、古いものの削除) に従って、AI エージェントのクライアント シークレットが正常にローテーションされました。
